## 1. What Is CLAUDE.md

CLAUDE.md is Claude Code's **project context file** — it tells Claude "what this project is, how to run it, and what rules to follow." It is **automatically loaded at the start of every conversation**, essentially acting as a system prompt prepended to every session.

### Core Characteristics

| Property | Description |
|----------|-------------|
| **Auto-loaded** | No manual reference needed — Claude Code reads it on startup |
| **Session-wide impact** | All instructions in the session are subject to its constraints |
| **Static context** | Unlike Skills, which are triggered on demand, CLAUDE.md is always in context |
| **Cumulative cost** | Every additional line consumes more context window in every conversation |

### Equivalents in Other AI Coding Tools

| Claude Code | Cursor | Windsurf | GitHub Copilot |
|------------|--------|----------|----------------|
| `CLAUDE.md` | `.cursorrules` | `.windsurfrules` | `.github/copilot-instructions.md` |

## 2. The Right Way to Start from Scratch

> **CLAUDE.md is not something you write upfront — it's something you grow through usage.**

### Why `/init` Auto-Generation is Not Recommended

Claude Code provides the `/init` command, which lets Claude analyze your codebase and auto-generate a CLAUDE.md. Sounds convenient, but research shows (see [Section 6](#6-academic-research-the-real-impact-of-context-files)) that **LLM-generated context files decrease task success rate by 2-3% and increase inference cost by 20%+**.

The reason is simple: out of 200 lines generated by `/init`, perhaps only 10 contain information Claude truly cannot infer from the code. The remaining 190 lines are redundant — things Claude can already figure out from `package.json`, file structure, and the code itself. This redundancy wastes the context window and interferes with Claude's judgment.

### The Right Starting Process

```text
Step 1: Don't write anything yet
  ↓ Start using Claude Code for your tasks directly
Step 2: Observe problems
  ↓ It used npm when you need pnpm? Note it.
  ↓ It doesn't know your test command? Note it.
  ↓ It edited generated files that shouldn't be touched? Note it.
Step 3: Add rules targeting specific problems
  ↓ Each time, add only the minimum content that solves the concrete issue
  ↓ One rule per problem — don't preemptively stuff in a bunch
Step 4: Organize after accumulation
  ↓ Distribute using the three-layer architecture (see Section 9)
  ↓ Convert anything that can be a Hook/Linter rule
```

### Comparison

| Approach | Day One Content | Precision | Long-term Maintenance |
|----------|----------------|-----------|----------------------|
| `/init` auto-generation | 200 lines | Low (90% redundant) | Hard (unclear what's useful) |
| Manual accumulation from real problems | 0 lines | High (every line solves a real problem) | Easy (every rule has a reason to exist) |

**This process is slower than `/init`, but every single line is something you actually need.**

## 3. Loading Mechanism and Priority

### Loading Locations (highest to lowest)

```text
~/.claude/CLAUDE.md          ← Global: shared across all projects (personal preferences, universal rules)
├── /project/CLAUDE.md       ← Project root: project-level settings
├── /project/src/CLAUDE.md   ← Subdirectory: module-level overrides
└── /project/.claude/rules/  ← Rules directory: modular rule management
```

### Loading Rules

1. **Global file** (`~/.claude/CLAUDE.md`): Always loaded — ideal for personal preferences
2. **Project root** (`/project/CLAUDE.md`): Loaded when working in that project
3. **Subdirectory files**: Only loaded when Claude accesses files in that directory
4. **Rules directory** (`.claude/rules/*.md`): All `.md` files are loaded — ideal for categorized management

### Multi-File Strategy: When to Split

| Scenario | Recommendation |
|----------|---------------|
| Content < 200 lines | A single `CLAUDE.md` is fine |
| Content > 200 lines | Split into `.claude/rules/` for modular management |
| Frontend / backend rules differ significantly | Place a `CLAUDE.md` in each subdirectory |
| Team vs personal preferences | Team rules go in `CLAUDE.md` (version controlled); personal preferences go in `~/.claude/CLAUDE.md` |

## 4. What to Write

**Core principle: Only write information that Claude cannot infer from the code itself.**

### 4.1 Build and Run Commands

The information Claude needs most but is least able to guess — how to run your project:

```markdown
## Development Commands

- Install dependencies: `pnpm install`
- Dev mode: `pnpm dev`
- Run tests: `pnpm test`
- Run a single test: `pnpm test -- --filter=<test-name>`
- Build: `pnpm build`
- Lint: `pnpm lint --fix`
```

### 4.2 Non-Obvious Architecture Decisions

Decisions about "why it's done this way" rather than "what was done":

```markdown
## Architecture Decisions

- Uses monorepo (pnpm workspaces); each subdirectory under packages/ is an independent package
- API layer uses tRPC — no REST endpoints
- State management uses Zustand, not Redux (team decision — do not change)
- Database migrations use Drizzle, not Prisma
```

### 4.3 Code Conventions (Non-Obvious Ones)

Only write rules that are not easily inferred from the code:

```markdown
## Conventions

- Component filenames use PascalCase; utility functions use camelCase
- All API responses must follow the `{ success, data, error }` structure
- Error codes are defined in `src/constants/errors.ts` — do not hardcode
- Commit message format: `type(scope): description`
```

### 4.4 Important Boundary Conditions

```markdown
## Important Notes

- Do not modify files under `src/generated/` — they are auto-generated by codegen
- Environment variables are defined in `.env.example` and must be kept in sync
- CI runs `pnpm typecheck` — type errors will block merges
```

## 5. What Not to Write

**Core principle: If the information can be obtained from the code, git history, or existing documentation, don't repeat it.**

### 5.1 Generic Project Descriptions

```markdown
<!-- Don't write this -->
## Project Overview
This is a frontend application built with React and TypeScript,
using the Next.js framework with Tailwind CSS for styling...
```

Why not? Claude can figure this out from `package.json` and the file structure.

### 5.2 General Language or Framework Knowledge

```markdown
<!-- Don't write this -->
## TypeScript Standards
- Use interface over type (unless unions are needed)
- Enable strict mode
- Avoid using any
```

Why not? This is TypeScript community consensus — Claude already knows it from training data. **Unless** your project intentionally deviates from convention (e.g., "we prefer `type` over `interface` here"), there's no need to write it.

### 5.3 Information Already Expressed in Code

```markdown
<!-- Don't write this -->
## Database Schema
The users table has id, name, email, created_at columns...
```

Why not? Claude can read the schema file directly, and the description in CLAUDE.md may become outdated.

### 5.4 Lengthy API Documentation

```markdown
<!-- Don't write this -->
## API Endpoints
GET /api/users - Get all users
POST /api/users - Create user
...(50 lines)
```

Why not? This information belongs in code comments or standalone documentation that Claude reads on demand. Loading 50 lines of API docs into every conversation is wasteful.

## 6. Academic Research: The Real Impact of Context Files

### Paper Summary

[Evaluating AGENTS.md](https://arxiv.org/html/2602.11988v1) (2026) is the first empirical study measuring how context files affect coding agents. The results are surprising:

| Configuration | Task Success Rate Change | Inference Cost Change |
|--------------|------------------------|-----------------------|
| LLM-generated context file | **Decreased 2-3%** | **Increased 20-23%** |
| Developer-written context file | Increased ~4% | Increased ~19% |
| No context file (baseline) | — | — |

### Why Can It Be Harmful?

1. **Information redundancy**: LLM-generated files highly overlap with existing documentation (README, docstrings), adding no new value
2. **Navigation interference**: Codebase overviews don't help agents find relevant files faster — they actually add 2-4 extra steps
3. **Sharp increase in inference cost**: Extra context forces the model to process more tokens without improving quality

### The Only Scenario Where It Helps

When a project has **absolutely no other documentation** (all markdown docs removed), an LLM-generated context file shows a positive effect (+2.7%), because it becomes the only documentation source.

### Implications for CLAUDE.md

| Approach | Effect |
|----------|--------|
| Auto-generate CLAUDE.md with an LLM | Most likely harmful — avoid |
| Copy README content into CLAUDE.md | Redundant — harmful |
| Only write what Claude cannot infer from code | Developer-written +4% — worth the investment |
| Minimal, specific, actionable instructions | Maximizes benefit, minimizes cost |

## 7. Writing Principles: Less is More

### Principle 1: Every Line Has a Cost

CLAUDE.md is loaded in **every conversation**. A 100-line CLAUDE.md means you pre-consume ~1000 tokens of context window per conversation. Ask yourself: "Is this line worth appearing in every conversation?"

### Principle 2: Specific Instructions > Vague Descriptions

```markdown
<!-- Bad: vague description -->
This project uses a monorepo architecture; please follow relevant best practices.

<!-- Good: specific instruction -->
Run tests with `pnpm --filter=<package> test`, not `pnpm test` (which runs all packages).
```

### Principle 3: Constraints > Suggestions

Claude responds better to explicit constraints ("don't do X") than vague suggestions ("try to do Y"):

```markdown
<!-- Bad: vague suggestion -->
Try to keep code concise.

<!-- Good: explicit constraint -->
Do not call fetch directly in components — all API requests must go through src/lib/api.ts.
```

### Principle 4: Maintenance Discipline

CLAUDE.md content becomes stale as the project evolves. Outdated instructions are worse than no instructions — they cause Claude to make incorrect decisions.

- Update CLAUDE.md after every major architecture change
- Periodically review: delete rules that no longer apply
- If a rule is already enforced by code (linter, CI), remove it from CLAUDE.md

## 8. Deterministic Execution: Replace Instructions with Hooks

> Source: [Matt Pocock — How to Force Claude Code to Use the Right CLI](https://www.youtube.com/watch?v=3CSi8QAoN-s)

This is the most overlooked optimization: **rules that can be enforced deterministically (Hooks, Linters, CI) should not live in CLAUDE.md.**

### The Problem: CLAUDE.md Instructions Are Non-Deterministic

```markdown
# CLAUDE.md
- Use pnpm, not npm
- Do not run git push
```

These instructions are merely "suggestions" — Claude will follow them most of the time, but not always. Meanwhile, they permanently consume the instruction budget, even when most conversations don't involve installing packages or git operations.

### The Solution: Convert to Hooks

Claude Code Hooks run deterministic scripts before/after tool execution and can **actually block** commands while guiding Claude to auto-correct:

```bash
# .claude/hooks/block-npm.sh
#!/bin/bash
if [[ "$TOOL_INPUT" =~ ^npm ]]; then
  echo "Use pnpm, not npm"
  exit 2  # exit 2 = block and feed error message back to Claude
fi
```

```jsonc
// .claude/settings.json
{
  "hooks": {
    "pre-tool-use": [
      {
        "matcher": { "tool_name": "bash" },
        "command": ".claude/hooks/block-npm.sh"
      }
    ]
  }
}
```

**Result**: Claude tries to run `npm install react-query` → Hook blocks it → Claude automatically switches to `pnpm install react-query`. Fully deterministic, zero instruction budget cost.

### Decision Flow: Instruction vs Hook vs Linter

```text
Can this rule be enforced deterministically by a machine?
├── Yes: CLI command preference (npm→pnpm) → Convert to Hook
├── Yes: Code style (ban positional params) → Convert to ESLint rule
├── Yes: Block dangerous operations (git push) → Convert to Hook
└── No: Architecture decisions, contextual information → Keep in CLAUDE.md
```

### Commonly Convertible Rules

| CLAUDE.md Instruction | Conversion Target | Method |
|-----------------------|-------------------|--------|
| "Use pnpm, not npm" | Hook | `pre-tool-use` intercepts bash commands |
| "Don't git push" | Hook | `pre-tool-use` intercepts bash commands |
| "Use our wrapper script for tests" | Hook | Intercept `npx jest` → run `./scripts/test.sh` instead |
| "No more than 3 function parameters" | ESLint | `max-params` rule |
| "Don't use any" | TSConfig | `"strict": true` + `@typescript-eslint/no-explicit-any` |
| "Sort imports" | ESLint | `eslint-plugin-import` |

### Core Philosophy

> **"Free up your instruction budget by enforcing rules deterministically. That way, the instructions remaining in CLAUDE.md are the ones that truly require LLM reasoning."**
> — Matt Pocock

This aligns with the research findings in Section 6: the less content in CLAUDE.md and the more precise it is, the better it performs. Move rules that Hooks and Linters can handle out, and let CLAUDE.md contain only information Claude genuinely needs to "understand."

## 9. The Three-Layer Architecture Model

Not all rules belong in CLAUDE.md. Based on **how a rule is enforced**, it should be assigned to the appropriate layer:

### Layer 1: Enforcement Layer (Deterministic)

| What Goes Here | Why |
|---------------|-----|
| Code formatting rules → ESLint / Prettier | Claude may forget rules, but code never forgets |
| Type checking → TypeScript strict mode | The compiler is more reliable than instructions |
| Test coverage → CI/CD pipeline | Automated verification, not dependent on LLM memory |
| CLI preferences → Hooks (see Section 8) | Deterministic interception, zero instruction budget cost |

**Principle**: Anything that can be enforced by code should not be expressed in natural language instructions.

### Layer 2: High-Frequency Recall Layer (CLAUDE.md Core)

| What Goes Here | Size Limit |
|---------------|------------|
| Project architecture overview (concise version) | Under 100-200 lines total |
| Key tech stack versions | |
| Core coding conventions (5 or fewer) | |
| Build and test commands | |

**Principle**: Content here is loaded in every conversation, so every line must be worth that cost.

### Layer 3: On-Demand Reference Layer (Reference Docs)

| What Goes Here | Location |
|---------------|----------|
| Detailed architecture documentation | `docs/architecture.md` |
| API reference | `docs/api.md` |
| Historical decision records | `docs/decisions/` |

**Key**: Use links to reference, don't stuff everything into CLAUDE.md:

```markdown
# In CLAUDE.md, write it like this
## Architecture
See `docs/architecture.md` for project architecture details — consult it before modifying API endpoints.
```

### Three-Layer Overview

```text
Layer 1: Enforcement (ESLint, TypeScript, CI/CD, Hooks)
  ↓ Code auto-validates; Claude gets immediate feedback on mistakes
Layer 2: High-frequency recall (CLAUDE.md core, <200 lines)
  ↓ Auto-loaded every conversation; only truly high-frequency, non-automatable info
Layer 3: On-demand reference (docs/, standalone .md files)
  ↓ Claude reads only when needed; referenced via links, not inlined
```

### Decision Flow

```text
Something you need to tell Claude
├── Can it be enforced by code?
│   └── Yes → Write as a linting rule / test / Hook (Layer 1)
├── Is it needed in every conversation?
│   └── Yes → Put in CLAUDE.md core (Layer 2)
└── Needed occasionally?
    └── Yes → Put in reference docs, link to it (Layer 3)
```

## 10. Starter Templates

### Minimum Viable Template (suitable for most projects)

```markdown
# CLAUDE.md

## Development Commands

- Install: `pnpm install`
- Dev: `pnpm dev`
- Test: `pnpm test`
- Build: `pnpm build`

## Important Constraints

- Do not modify files under `src/generated/`
- All API requests must go through `src/lib/api.ts`
- Commit message format: `type(scope): description`
```

That's enough. Under 30 lines, containing only what Claude actually needs.

### Mid-Size Project Template

```markdown
# CLAUDE.md

## Development Commands

- Install: `pnpm install`
- Dev: `pnpm dev` (starts Next.js dev server + API)
- Test: `pnpm test` (Jest + React Testing Library)
- Single test: `pnpm test -- --testPathPattern=<pattern>`
- Type check: `pnpm typecheck`
- Lint: `pnpm lint --fix`

## Architecture

- Monorepo: `packages/web` (frontend), `packages/api` (backend), `packages/shared` (shared types)
- Database: PostgreSQL + Drizzle ORM; migrations in `packages/api/drizzle/`
- State management: Zustand (not Redux)
- API communication: tRPC (no REST)

## Constraints

- Do not modify `packages/shared/generated/` — generated by codegen
- New API endpoints must go under `packages/api/src/routers/`
- Frontend components go in `packages/web/src/components/`, organized by feature
- All environment variables are defined in `.env.example` — keep in sync when adding new ones
- CI runs typecheck + lint + test; all must pass before merging
```

### Global Template (`~/.claude/CLAUDE.md`)

```markdown
# Global Preferences

- Respond in English
- Write commit messages in English
- Prefer function declarations over arrow functions
- Don't add a summary at the end of responses
```

## 11. Advanced Patterns

### 11.1 Rules Directory for Modular Management

When CLAUDE.md exceeds 200 lines, split into `.claude/rules/`:

```text
.claude/rules/
├── coding-style.md    # Code style
├── testing.md         # Testing standards
├── security.md        # Security guidelines
├── workflow.md        # Workflow rules
└── pragmatism.md      # Exception handling rules
```

All `.claude/rules/*.md` files are auto-loaded — no need to reference them in CLAUDE.md.

**Note**: Splitting does not save context — everything is still fully loaded. The benefit is **readability and maintainability**.

### 11.2 Subdirectory Overrides

Set different rules for different modules in a monorepo:

```text
/project/CLAUDE.md                  # Shared rules
/project/packages/web/CLAUDE.md     # Frontend-specific rules
/project/packages/api/CLAUDE.md     # Backend-specific rules
```

Subdirectory CLAUDE.md files are only loaded when Claude accesses files in that directory.

### 11.3 Pairing with Skills

CLAUDE.md holds context that is **always needed**; Skills hold instructions that are **triggered on demand**:

```markdown
# Referencing a skill in CLAUDE.md
## Code Review
Use `/review` to trigger the code review workflow (see .claude/skills/review/)
```

### 11.4 Handling Sensitive Information

```markdown
<!-- Don't put this in CLAUDE.md -->
API_KEY=sk-xxxxx

<!-- Write this instead -->
## Environment Variables
All API keys are defined in `.env` (added to .gitignore).
See `.env.example` for the required variables.
```

## 12. Common Anti-Patterns

### Anti-Pattern 1: The Encyclopedia

```markdown
<!-- Anti-pattern: stuffing all knowledge in -->
# CLAUDE.md (800+ lines)
## Project History
This project started in 2023...
## Complete API Documentation
### GET /api/users
### POST /api/users
### PUT /api/users/:id
...(200 lines)
## TypeScript Style Guide
...(150 lines)
## Git Workflow
...(100 lines)
```

**Problem**: Wastes massive context every conversation, and most content goes unused.

**Fix**: Slim down to under 50 lines; move details to standalone docs or Skills.

### Anti-Pattern 2: LLM-Generated

```markdown
<!-- Anti-pattern: letting Claude generate CLAUDE.md -->
"Claude, analyze this codebase and generate a CLAUDE.md"
```

**Problem**: Research shows LLM-generated context files decrease success rate by 2-3% and increase cost by 20%+. The generated content is mostly information Claude can already infer from code.

**Fix**: Write it manually — only include what Claude cannot infer.

### Anti-Pattern 3: Stale and Neglected

```markdown
<!-- Anti-pattern: not updated in six months -->
## Tech Stack
- Uses Webpack (switched to Vite three months ago)
- Tests with Jest (switched to Vitest)
- Deploys to Heroku (migrated to Vercel)
```

**Problem**: Outdated instructions are worse than no instructions — Claude will follow incorrect guidance.

**Fix**: Update in sync with architecture changes; periodically review and remove stale content.

### Anti-Pattern 4: README Duplication

```markdown
<!-- Anti-pattern: copying README content -->
# CLAUDE.md
## About This Project
(Identical content to README.md...)
```

**Problem**: Information redundancy increases context consumption without adding value. Claude can read the README on its own.

**Fix**: CLAUDE.md and README serve different purposes — README is written for humans, CLAUDE.md is written for Claude.

### Anti-Pattern 5: Letting Claude Maintain CLAUDE.md

```markdown
<!-- Anti-pattern: letting Claude update CLAUDE.md -->
"Claude, update CLAUDE.md based on our conversation"
```

**Problem**: Claude tends to stuff everything it learned into the file, causing CLAUDE.md to bloat out of control.

**Fix**: CLAUDE.md should be maintained by humans. If you need automatic correction capture, use a dedicated tool like [claude-reflect](https://github.com/BayramAnnakov/claude-reflect), which includes a human review stage before writing.

> **gingersnap's experience**: "Let Claude write code, but never let Claude touch CLAUDE.md."

## 13. Maintenance Practices

CLAUDE.md is not a write-once-and-forget file. Without maintenance discipline, it decays rapidly.

### 13.1 Regular Cleanup

- **Weekly review** of CLAUDE.md content
- Delete outdated and duplicated content
- Confirm each rule is still valid and necessary

### 13.2 Layered Management

```text
CLAUDE.md              # Core rules, <200 lines
├── docs/arch.md       # Architecture docs (Layer 3)
├── docs/api.md        # API reference (Layer 3)
└── .cursorrules       # Editor rules (if also using Cursor)
```

### 13.3 Version Control

CLAUDE.md **must be tracked in git**:

- Team members can review changes to CLAUDE.md
- You can trace when rules were added or removed
- Prevents personal preferences from polluting the team's shared CLAUDE.md (personal preferences go in `~/.claude/CLAUDE.md`)

### 13.4 Better Alternatives

| Approach | Pros | Cons |
|----------|------|------|
| **Periodic retrospectives** | Human-curated, high quality | Requires discipline |
| **Linting-first** | Enforced deterministically | Only applicable to formatting issues |
| **Session log analysis** | Comprehensive | Information overload |
| **claude-reflect** | Auto-captures corrections, writes after review | Requires additional setup |

## 14. Common Pain Points and Solutions

### Pain Point 1: Claude Doesn't Consult Reference Docs

> "My problem is that Claude often doesn't look up the reference documentation."

**Solution**:

1. In CLAUDE.md, **explicitly state when to consult** using specific trigger conditions:

```markdown
## Reference Documentation
- Before modifying API endpoints, you **must** read `docs/api.md` first
- Before modifying database schema, you **must** read `docs/database.md` first
```

2. Use precise trigger conditions (not vague "see docs/")
3. Consider wrapping reference docs as Claude Skills, leveraging `description` for automatic triggering

### Pain Point 2: CLAUDE.md Keeps Growing

**Solution**: Apply the three-layer architecture model (Section 9) and distribute content by layer:

- Can be automated → Layer 1 (Hooks / Linter)
- High-frequency essential → Layer 2 (CLAUDE.md, keep under 200 lines)
- Occasionally needed → Layer 3 (standalone docs, link to them)

### Pain Point 3: Not Sure What to Write

**Solution**: Don't write anything yet. Observe where Claude makes mistakes or deviates from your expectations, and **only add rules when problems actually occur**. This is more effective than guessing what Claude needs upfront.

## 15. CLAUDE.md vs Skills vs Subagent: Division of Responsibility

| Dimension | CLAUDE.md | Skills | Subagent |
|-----------|-----------|--------|----------|
| **When loaded** | Auto-loaded every conversation | Loaded on demand when triggered | Launched when delegated |
| **Best for** | Build commands, architecture decisions, global constraints | Specific workflows, step-by-step instructions | Independent research or analysis tasks |
| **Context consumption** | Permanently occupied | Occupied only when triggered | Uses a separate context window |
| **Update frequency** | On architecture changes | On workflow changes | On requirement changes |
| **Maintained by** | Entire team | Workflow owner | Task owner |

### Decision Flow

```text
Does this information need to appear in every conversation?
├── Yes → Put in CLAUDE.md (but confirm it's truly needed every time)
└── No → Is this a repeatable workflow?
    ├── Yes → Write as a Skill (/skill-name to trigger)
    └── No → Does this task need extensive context and can be completed independently?
        ├── Yes → Write as a Subagent
        └── No → Probably doesn't need to be persisted — just tell Claude verbally
```

## 17. Quick Checklist

When writing or reviewing CLAUDE.md, verify each item:

- [ ] **Line count < 100**: If exceeded, consider splitting into `.claude/rules/` or moving to Skills
- [ ] **No redundancy**: Nothing duplicated from README, package.json, or code
- [ ] **Not LLM-generated**: Written by a human, not auto-generated by AI
- [ ] **Has build commands**: At minimum includes install, dev, test, and build commands
- [ ] **Has explicit constraints**: Uses "don't" and "must" instead of "try to" or "it's recommended"
- [ ] **No stale information**: All mentioned tools, paths, and commands are currently valid
